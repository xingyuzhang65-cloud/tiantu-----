## 2. 背景与问题

跨境物流业务中，一票头程货物到达海外仓后，并不一定一次性全部发往最终目的地。常见场景包括：

- 同一票头程运单包含多个 FBA 箱号，需要按仓库、目的地、客户要求或尾程服务分批出库；
- 货物已入海外仓，但客户尚未确认全部出库，货物需要暂存并持续产生仓租；
- 部分箱子需要转运，部分箱子继续留仓，母单库存和子单出库数量必须实时对应；
- 海外仓完成拣货后，仓库需要回填转单号、承运商、箱号和交接信息；
- 尾程运输过程还会发生取消、回退、异常、签收和 POD 回传。

如果“海外仓暂存”和“海外中转单”只是两个彼此独立的列表，会产生以下问题：

1. 暂存库存已经被下单，但列表仍显示可用；
2. 同一个箱号被重复拆到多个子单；
3. 子单取消后，箱号没有回流，或重复回流导致库存虚增；
4. 子单状态改变后，母单仍停留在错误状态；
5. 客户、仓库、操作员、财务看到的数量和状态不一致；
6. 一个头程运单拆出多个尾程运单后，无法通过链路追溯责任、费用和签收结果。

因此，本产品需要将两个界面定义为同一业务链路上的两个工作台：

- 海外仓暂存：以母单库存和货箱可用性为中心，负责选择可出库箱、填写收件地址、添加操作指令并创建子单；
- 海外中转单：以子单履约状态为中心，负责确认、下单、出运、签收、转单号维护、费用、附件和操作日志。

## 3. 产品目标与非目标

### 3.1 产品目标

1. 建立母单、箱号、子单、转单号之间的唯一关联关系。
2. 支持一票多箱、多批次、部分出库和多子单拆分。
3. 保证“创建子单”和“扣减可用箱”在业务上原子完成。
4. 让每一个子单都能沿着头程运单追溯到具体箱号，并能追踪到承运商和签收凭证。
5. 通过明确的状态机控制操作权限，降低误下单、重复下单和错发风险。
6. 为仓库、销售、跟单、财务和客服提供同一套可审计的数据。

### 3.2 非目标

- 本期不重做头程下单、报关、清关和干线运输系统；
- 本期不定义海外仓实际库位、库内作业 PDA 或 WMS 货架算法；
- 本期不替代承运商轨迹系统，只定义轨迹接入和状态映射；
- 本期不把费用结算规则全部产品化，先支持费用明细、计费依据和对账接口；
- 本期不把所有国家和服务的业务规则一次性固化，优先支持美国 FBA、Walmart、TikTok 和私人地址。

## 4. 核心概念与对象关系

### 4.1 核心对象定义

| 对象 | 定义 | 业务主键 | 主要负责人 |
| --- | --- | --- | --- |
| 头程运单／母单 | 国内发往海外仓的一票货物，是库存和箱号的来源 | headWaybillNo | 头程／订单系统 |
| 海外仓暂存单 | 母单到达海外仓后的库存承载对象，记录总件数、库存件数、可用件数和库龄 | storageOrderId，通常与 headWaybillNo 关联 | 海外仓运营 |
| 货箱 | 母单下可被独立分配的最小出库单元，通常对应 FBA/IBR 箱号 | boxId 或 fbaBoxNo | 海外仓／仓库 |
| 海外中转子单 | 从一个母单中选出一批箱后生成的独立尾程履约单 | transitOrderId | 中转运营 |
| 转单号 | 承运商或尾程服务商实际承运的运单号，一个子单可有一个主转单号，也可按箱有明细转单号 | transferNo | 仓库／承运商 |
| 轨迹事件 | 承运商或操作员产生的运输节点，如已出运、转运中、签收 | eventId | 轨迹系统 |
| 操作指令 | 仓库需要执行的增值服务或仓储操作，如免仓期、拦截、贴标、拍照 | instructionId | 仓库运营 |
| 费用明细 | 操作指令或中转报价对应的计费项 | feeId | 财务／运营 |
| 附件 | 交接单、装箱单、POD、异常照片等凭证 | attachmentId | 仓库／客服 |

### 4.2 对象关系

业务关系为：

头程运单 1 ── N 货箱  
海外仓暂存单 1 ── N 货箱  
海外中转子单 1 ── N 货箱  
头程运单 1 ── N 海外中转子单  
海外中转子单 1 ── N 转单号明细  
海外中转子单 1 ── N 轨迹事件、操作指令、费用明细、附件

关键原则：

1. 箱号是拆单和库存扣减的最小粒度，不能只按件数做虚拟扣减。
2. 子单必须保存 parentHeadWaybillNo，不能只保存客户或 FBA 号。
3. 同一箱号在同一时刻只能属于一个有效的未取消子单。
4. 取消子单时，箱号回流的是原母单的同一箱号，而不是只回写一个数量。
5. 一个母单可以在不同时间生成多个子单，orderSeq 用于标识同一母单下的批次顺序。

## 5. 两个界面的产品定位

### 5.1 海外仓暂存界面

定位为“库存池与出库申请工作台”，关注：

- 哪些箱已到仓；
- 哪些箱仍可用；
- 哪些箱已经被子单占用；
- 哪些箱已完成出库；
- 当前库龄和仓租情况；
- 选择箱号、填写目的地址和仓库操作要求；
- 创建一张待确认的海外中转子单。

当前列表支持的业务状态：

| 状态 | 含义 | 进入条件 | 主要动作 |
| --- | --- | --- | --- |
| 运输中 | 头程货物尚未完成海外仓入库确认 | 头程已发运 | 查看、等待入仓 |
| 暂存 | 已进入海外仓，箱号仍有可用库存 | 入仓完成且存在可用箱 | 下单、查看、日志、导出 |
| 暂存已完成 | 母单下可用箱全部出库或完成释放 | 可用箱为 0 且出库闭环 | 查看完成详情、导出、日志 |

本页面的关键动作不是直接完成尾程运输，而是创建子单。提交后，页面必须显示：

- 已选择箱号；
- 生成的子单号；
- 子单状态为待确认；
- 母单剩余总件数、库存件数、可用件数；
- 已拆分批次数和最近操作时间。

### 5.2 海外中转单界面

定位为“子单履约和运输执行工作台”，关注：

- 哪些子单等待业务确认；
- 哪些子单已确认可下单；
- 哪些子单已完成下单；
- 哪些子单已经出运；
- 哪些子单已签收或取消；
- 每个子单对应的箱号、客户、仓库、目的地、服务和转单号；
- 维护费用、附件、操作指令和业务日志。

当前主要状态页签：

待确认、已确认、已下单、转运中、签收、取消。

该页面不再承担“选择母单库存”的职责。子单的箱号来源必须来自暂存页已经锁定的箱号，不能在子单页面自由输入不存在的箱号。

### 5.3 信息归属原则

| 信息 | 暂存页 | 中转单页 | 主数据归属 |
| --- | --- | --- | --- |
| 母单总件数／库存件数／可用件数 | 展示、实时扣减 | 只读引用 | 海外仓库存域 |
| 货箱号 | 选择、锁定、回流 | 只读展示、按箱维护转单号 | 箱号库存域 |
| 收件地址 | 首次填写、校验 | 待确认／已确认阶段可编辑 | 子单域 |
| 客户备注、内部备注 | 创建时带入 | 查看与审计 | 子单域 |
| 转单号、承运商 | 不负责最终维护 | 维护、校验、回传 | 运输履约域 |
| 费用与操作指令 | 创建时选择仓储指令 | 维护报价、服务费和日志 | 费用域 |
| 轨迹、签收、POD | 不维护 | 接收并展示 | 轨迹／凭证域 |

## 7. 业务流程总览

### 7.1 主流程

1. 头程运单创建并发运，产生母单和货箱清单。
2. 海外仓接收入库，更新母单总件数、库存件数、可用件数和入仓时间。
3. 海外仓暂存页展示可用母单和箱号。
4. 操作员进入暂存单详情，选择一个或多个可用箱。
5. 填写 FBA/Walmart/TikTok/私人地址、仓库代码、邮编、城市、州、地址详情和备注。
6. 选择仓库操作指令，系统校验箱号和地址必填项。
7. 生成海外中转子单，状态为待确认；原箱号立即标记为已分配。
8. 子单进入海外中转单的待确认页签。
9. 业务员或跟单确认子单，状态变为已确认。
10. 运输运营完成下单，状态变为已下单，并维护承运商和转单号。
11. 仓库完成交接并点击出运，状态变为转运中。
12. 轨迹系统或运营回传签收，状态变为签收并挂接 POD。
13. 母单在所有可用箱闭环后变为暂存已完成。

### 7.2 流程示意

头程运单  
→ 海外仓入库  
→ 母单／暂存库存  
→ 选择可用箱  
→ 生成子单（待确认）  
→ 已确认  
→ 已下单＋转单号  
→ 转运中  
→ 签收＋POD  
→ 母单库存释放／暂存完成

## 8. 数据流设计

### 8.1 数据源与责任边界

| 数据域 | 上游来源 | 本产品处理 | 下游消费 |
| --- | --- | --- | --- |
| 头程运单 | 订单／头程系统 | 只读关联、补充目的仓信息 | 暂存页 |
| 入仓和库存 | 海外仓/WMS | 记录快照、按箱扣减和回流 | 暂存页、报表 |
| 箱号 | WMS/FBA 箱清单 | 锁定、拆分、回流、追踪 | 子单、轨迹 |
| 地址 | 客户资料／运营录入 | 校验、补全、版本化 | 仓库、承运商 |
| 子单 | 本产品创建 | 状态机、操作审计 | 运输、客服、财务 |
| 转单号 | 承运商／仓库 | 绑定箱号和服务 | 轨迹系统、客户 |
| 轨迹 | 承运商接口 | 映射、去重、展示 | 子单、客服 |
| 费用 | 运营报价／仓储规则 | 费用明细、审核、对账 | 财务 |

### 8.2 入仓数据流

1. 头程系统推送 headWaybillNo、客户、FBA 号、预报箱数和服务。
2. WMS 接收入库，生成或确认每个 fbaBoxNo。
3. WMS 回传实收箱数、重量、体积、入仓时间、仓库代码和异常状态。
4. 本产品建立 storageOrder 和 storageBox 记录。
5. 系统计算：
   - totalCount：母单预报或实收总箱数；
   - inventoryCount：已入仓且未出库的箱数；
   - availableCount：状态为可分配的箱数；
   - allocatedCount：已被有效子单锁定但尚未完成履约的箱数；
   - releasedCount：已出库或已签收的箱数。
6. 入仓事件必须幂等，重复回传不能重复增加库存。

### 8.3 创建子单数据流

创建子单是整个链路中最重要的事务，必须同时完成以下动作：

1. 校验母单存在且属于当前仓库权限；
2. 校验所有选中箱号属于该母单；
3. 校验所有箱号当前状态为 available；
4. 校验请求中的地址版本、仓库代码和服务有效；
5. 校验没有重复的幂等请求号；
6. 创建 transitOrder 主记录；
7. 创建 transitOrderBox 关联记录；
8. 将箱号从 available 更新为 allocated；
9. 更新母单 availableCount、allocatedCount；
10. 写入 storage.box.allocated 和 transit.order.created 事件；
11. 返回子单号、母单剩余可用数和箱号明细。

以上动作必须在一个数据库事务内完成。任一动作失败，全部回滚。

### 8.4 子单履约数据流

子单状态变化时，不应只改一个 status 字段，还应记录：

- previousStatus；
- currentStatus；
- action；
- operator；
- operatedAt；
- source；
- reason；
- relatedTransferNo；
- relatedBoxIds；
- requestId。

状态事件示例：

| 事件 | 产生时机 | 事件载荷 | 下游动作 |
| --- | --- | --- | --- |
| storage.box.allocated | 子单创建成功 | 母单、子单、箱号、操作人 | 刷新暂存库存 |
| transit.order.created | 子单创建成功 | 子单全量快照 | 生成待确认任务 |
| transit.order.confirmed | 已确认 | 子单号、确认人 | 允许下单 |
| transit.order.ordered | 已下单 | 子单号、服务、转单号 | 通知仓库和承运商 |
| transit.order.shipped | 出运 | 子单号、箱号、出运时间 | 开始轨迹追踪 |
| transit.order.signed | 签收 | 子单号、签收时间、POD | 释放履约占用、完成结算 |
| transit.order.cancelled | 取消 | 子单号、箱号、原因 | 箱号回流可用库存 |
| storage.box.released | 取消或回退 | 母单、箱号、原因 | 恢复可用数并触发对账 |

### 8.5 库存数量计算

推荐的生产计算公式：

- totalCount = 预报箱数或最终实收箱数；
- inventoryCount = availableCount + allocatedCount + inWarehouseExceptionCount；
- availableCount = 可分配箱数；
- allocatedCount = 已分配给未完成子单的箱数；
- releasedCount = 已出库、已签收或已取消后不再回到当前库存的箱数。

对于取消子单：

availableCount_new = availableCount_old + canceledBoxCount

对于创建子单：

availableCount_new = availableCount_old - selectedBoxCount

严禁仅依据前端传入的 count 直接加减，必须依据箱号集合做差集校验。

## 9. 运单流设计

### 9.1 运单编号体系

| 编号 | 示例 | 生成方 | 生命周期 | 是否唯一 |
| --- | --- | --- | --- | --- |
| 头程运单号 | HD2606161063／USSZAS2508261001 | 头程系统 | 从头程下单到入仓完成 | 全局唯一 |
| 海外仓运单号 | 由海外仓或中转系统生成 | 海外仓 | 入仓和仓内处理 | 全局唯一 |
| 子单号 | parentHeadWaybillNo_日期_序号 | 本产品 | 从创建到履约完成 | 全局唯一 |
| 转单号 | 1Z0VV966030991、TRK26070000 | 承运商／仓库 | 下单后到签收 | 服务商范围唯一 |
| FBA／IBR 箱号 | FBA19DNZH02MU000318 | 亚马逊或客户 | 货箱全生命周期 | 母单内唯一，建议全局去重 |
| referenceId | REF-FBA-00001 | 本产品／承运商 | 轨迹和对账 | 业务范围唯一 |

### 9.2 一票拆多单示例

假设头程母单 HD001 包含 6 个箱：

- 子单 T001：箱 1、箱 2，目的仓 ONT8，服务 A；
- 子单 T002：箱 3、箱 4、箱 5，目的仓 ONT8，服务 B；
- 箱 6：继续留在海外仓暂存。

创建 T001 后：

- HD001.availableCount 从 6 变为 4；
- box1、box2 状态变为 allocated；
- T001.status = 待确认。

创建 T002 后：

- HD001.availableCount 从 4 变为 1；
- box3、box4、box5 状态变为 allocated；
- T002.status = 待确认。

T001 签收后：

- box1、box2 状态变为 delivered；
- T001.status = 签收；
- HD001 仍有箱 6 可用，不得标记为暂存已完成。

T002 取消后：

- box3、box4、box5 回流为 available；
- HD001.availableCount 从 1 变为 4；
- T002.status = 取消；
- 箱号不能回到其他母单。

### 9.3 运单状态机

#### 子单正向状态

待确认 → 已确认 → 已下单 → 转运中 → 签收

#### 可逆状态

- 待确认 → 取消：业务确认创建错误、客户取消或地址资料无效；
- 已确认 → 取消：二次确认后仍可取消；
- 已下单 → 已确认：仅限有权限的回退操作；
- 转运中 → 已下单：仅限轨迹纠错或出运撤销；
- 签收 → 转运中：仅限签收误操作纠正，必须二次确认并留痕。

#### 不允许的跳转

- 待确认不可直接转运中；
- 已确认不可直接签收；
- 取消不可直接恢复为已确认，必须通过新子单重新选择可用箱；
- 已签收的箱不能再次被新子单选择；
- 不是同一母单的箱不能拼入同一子单，除非后续产品明确支持跨母单合单。

### 9.4 状态转换矩阵

| 当前状态 | 操作 | 目标状态 | 前置条件 | 库存影响 |
| --- | --- | --- | --- | --- |
| 暂存 | 选择箱并提交 | 待确认 | 箱号可用、地址完整 | available 减少，allocated 增加 |
| 待确认 | 确认 | 已确认 | 资料完整、未发生冲突 | 无 |
| 待确认 | 取消 | 取消 | 有取消原因 | 箱号回流 available |
| 已确认 | 下单确定 | 已下单 | 服务和承运商有效 | 无 |
| 已确认 | 取消 | 取消 | 二次确认 | 箱号回流 available |
| 已下单 | 出运 | 转运中 | 转单号、承运商、箱号齐全 | allocated 保持，履约开始 |
| 已下单 | 回退 | 已确认 | 有权限、填写回退原因 | 无 |
| 转运中 | 签收 | 签收 | 签收事件或 POD 有效 | allocated 转 delivered |
| 转运中 | 回退 | 已下单 | 轨迹纠错权限 | 无 |
| 签收 | 回退 | 转运中 | 误签收处理权限 | delivered 暂不释放 |

### 9.5 转单号与箱号规则

1. 子单出运前必须至少有一个有效转单号。
2. 若承运商按箱出单，应维护箱号级 transferNo；若承运商只提供主单号，则主单号挂在子单，箱号明细继承主单号。
3. 同一转单号不能绑定到不同客户、不同目的地或不同服务，除非承运商明确支持合单。
4. 转单号变更必须保存旧值、新值、变更原因、操作人和时间。
5. 轨迹回传以 transferNo 为主键，以 childOrderId 和 boxId 为辅助关联。
6. 轨迹状态与业务状态分离存储：承运商的 In Transit、Delivered 不直接覆盖业务状态，需经过映射规则和人工纠错机制。

## 12. 详细运单流与数据流时序

### 12.1 正常时序

1. 头程系统创建母单 HD001 和箱号 B1、B2、B3。
2. WMS 回传入仓成功，母单状态为暂存，availableCount=3。
3. 用户在暂存页选择 B1、B2，填写 ONT8、92551 和地址。
4. 系统在事务内创建子单 T001，并锁定 B1、B2。
5. 暂存页收到刷新事件，availableCount=1。
6. 中转单页收到 transit.order.created，T001 进入待确认。
7. 跟单确认 T001，状态变为已确认。
8. 运输运营填写承运商和转单号，点击下单确定，状态变为已下单。
9. 仓库完成装车交接，点击出运，状态变为转运中。
10. 轨迹服务回传运输中节点，系统更新轨迹但不重复写业务状态。
11. 轨迹服务回传签收和 POD，系统更新 T001 为签收，B1、B2 为 delivered。
12. 用户在暂存页再次选择 B3 创建 T002，或母单在 B3 处理完后才变为暂存已完成。

### 12.2 取消时序

1. 用户在待确认或已确认页选择 T001，发起取消。
2. 系统弹出影响范围：箱号、件数、当前地址、当前费用和回流数量。
3. 用户填写取消原因并二次确认。
4. 系统锁定 T001，校验它没有进入已下单或转运中。
5. 更新 T001.status=取消。
6. B1、B2 状态回流 available。
7. 母单 availableCount 加 2。
8. 发送 transit.order.cancelled 和 storage.box.released。
9. 暂存页、子单页、日志页刷新为一致结果。

### 12.3 并发冲突时序

如果用户 A 和用户 B 同时选择同一箱号：

1. 暂存页只做短时前端勾选，不代表最终占用；
2. 提交时后端以 boxId 行锁或乐观锁校验；
3. 先成功者获得锁，后提交者返回“箱号已被子单 T001 占用”；
4. 后提交者不能创建部分成功的子单；
5. 前端刷新母单和箱号列表，并保留用户未提交的地址草稿。

## 15. 一致性、幂等与异常处理

### 15.1 一致性

创建子单、取消子单、状态变更必须使用状态机和事务。不能通过多个前端请求分别更新母单数量、箱号状态和子单状态。

### 15.2 幂等

- 创建子单以 requestId + parentHeadWaybillNo 作为幂等键；
- 轨迹事件以 carrier + transferNo + eventCode + occurredAt 做去重；
- 上传附件以文件 hash + 业务对象做重复检测；
- 取消、签收、回退重复点击只能返回当前结果，不能重复扣减或回流。

### 15.3 异常场景

| 场景 | 系统处理 |
| --- | --- |
| 箱号不存在 | 拒绝提交，返回具体箱号 |
| 箱号已被占用 | 返回占用子单和占用时间 |
| 地址缺失 | 按字段提示，不创建子单 |
| 私人地址无电话 | 阻止提交 |
| 承运商转单号重复 | 阻止出运并提示原绑定子单 |
| 轨迹回传顺序倒置 | 进入异常队列，不覆盖最新业务状态 |
| 取消时已开始出运 | 禁止自动取消，转人工异常处理 |
| WMS 回传重复入库 | 根据事件号幂等 |
| 子单创建超时 | 前端可用 requestId 查询创建结果，不允许盲目重提 |
| 母单库存为负 | 立即冻结母单，生成库存对账任务 |

## 18. 验收标准

### 18.1 暂存创建子单

- 选择 2 个可用箱后，创建成功的子单仅包含这 2 个箱；
- 母单可用数减少 2；
- 其余箱仍可被下一张子单选择；
- 子单状态为待确认；
- 刷新页面后关系仍存在；
- 重复点击不会生成两张子单。

### 18.2 取消与回流

- 待确认或已确认子单取消后，原箱号回到母单；
- 回流数量与实际箱号数量一致；
- 同一子单重复取消不会重复增加库存；
- 取消原因、操作者和箱号均可查日志。

### 18.3 运单履约

- 已确认才能下单；
- 已下单且具备承运商、转单号才能出运；
- 转运中才能签收；
- 签收后可查看 POD；
- 轨迹事件重复推送不会重复生成日志或改变状态。

### 18.4 数据一致性

- 母单箱号集合与子单箱号集合可相互反查；
- 所有有效子单箱号并集不超过母单箱号集合；
- 母单 availableCount 等于实时可用箱号数；
- 取消、回退、签收后的数量均可由箱号状态重算；
- 库存对账发现差异时可以定位到具体事件和操作人。

## 19. 当前实现与生产化差距

当前前端原型使用本地内存和事件订阅模拟链路：

- createdTransitChildOrders 作为子单内存集合；
- removedStorageBoxCounts 和 removedStorageBoxNumbers 作为母单扣减快照；
- subscribeOverseasTransitFlow 作为页面间刷新通知；
- 页面刷新或服务重启后数据会丢失；
- 目前没有真正的 WMS、承运商轨迹和持久化数据库。

生产化必须补齐：

1. 数据库事务和版本控制；
2. WMS 入库、箱号和出库接口；
3. 承运商下单、转单号和轨迹接口；
4. 消息队列与失败重试；
5. 权限、审计和数据脱敏；
6. 对账和人工异常处理台；
7. 文件存储、POD 防篡改和下载权限；
8. 监控、告警、链路追踪和数据留存策略。

## 20. 分阶段实施建议

### Phase 1：链路打通

- 持久化母单、箱号、子单；
- 完成创建子单、确认、取消、箱号回流；
- 页面状态和数量从后端读取；
- 建立操作日志。

### Phase 2：仓库与承运商协同

- 接入 WMS 出库确认；
- 接入承运商下单和转单号；
- 支持箱级转单号；
- 支持轨迹和 POD。

### Phase 3：费用与经营分析

- 费用规则、报价、审核和对账；
- 仓租、免仓期和超期预警；
- 转出率、时效、异常率和客户视图；
- 多仓、多国家、多服务扩展。
